1. APPLICATION OVERVIEW
================================================================================

Paperboy is a Rails 8 internal web application built for the County of Ventura
General Services Agency (GSA). It serves as a centralized forms platform that
handles:

  - Dynamic form creation and management (no-code form builder)
  - Form submission workflows with multi-step approval routing
  - Task inbox for supervisors/managers to approve, deny, or reassign
  - Employee submission tracking with status timelines
  - Report generation (CSV/PDF) with scheduling
  - Power BI dashboard embedding
  - Billing tools and invoice generation via MSSQL stored procedures
  - Access control at both group and organizational hierarchy levels
  - Help desk ticketing system
  - Lookup table management for reference data

The app is used by County of Ventura employees to submit various operational
forms (parking permits, probation transfer requests, critical incident reports,
OSHA forms, leave of absence, workplace violence reports, etc.) and by
managers/administrators to process those submissions.

Primary URL: gsa-forms (internal)
Module Name: FormsApp

3.3 MSSQL COLUMN NAMING CONVENTION
-----------------------------------

CRITICAL: Only 4 models use PascalCase MSSQL column names with alias_attribute
mappings. All other tables (35+) use standard snake_case columns.

The 4 PascalCase tables are:
  1. Employee (table: Employees)
  2. Group (table: Groups)
  3. EmployeeGroup (table: Employee_Groups)
  4. GroupPermission (table: Group_Permissions)

IMPORTANT: alias_attribute works with where(), order(), select(), find_by(),
exists?(), and group() — but pluck(:id) does NOT translate to the PK column.
You must use pluck(:GroupID) explicitly for PascalCase tables.

4. AUTHENTICATION & SESSION MANAGEMENT
================================================================================

4.1 AUTHENTICATION FLOW
------------------------

PRIMARY: OAuth via Microsoft Entra ID (Azure AD)
  Route: GET /auth/callback -> SessionsController#create_oauth
  Flow:
    1. User clicks login -> redirected to Entra ID
    2. Entra ID authenticates -> redirects back with OmniAuth callback
    3. SessionsController#create_oauth extracts email from callback
    4. Finds Employee record by email (EE_Email column)
    5. Populates session hash with employee data
    6. Redirects to root_path

FALLBACK: Legacy login for admin impersonation
  Route: POST /login -> SessionsController#create_legacy
  Flow:
    1. Admin enters employee ID or email
    2. SessionUser.authenticate() finds employee
    3. Session populated same as OAuth flow

LOGOUT:
  Route: DELETE /logout -> SessionsController#destroy
  Clears entire session and redirects to root

4.2 SESSION DATA STRUCTURE
--------------------------

session[:user] = {
  "employee_id"   => Integer,    # Employee.EmployeeID
  "email"         => String,     # Employee.EE_Email
  "first_name"    => String,     # Employee.First_Name
  "last_name"     => String,     # Employee.Last_Name
  "phone"         => String,     # Employee.Work_Phone
  "supervisor_id" => Integer,    # Employee.Supervisor_ID
  "department"    => String,     # Employee.Department
  "agency"        => String,     # Employee.Agency
  "unit"          => String      # Employee.Unit
}

session[:user_id] = Integer  # Quick-access copy of employee_id

4.3 CURRENT ATTRIBUTES
-----------------------

File: app/models/current.rb
Type: ActiveSupport::CurrentAttributes

Current.user stores the session[:user] hash for the duration of each request,
set by ApplicationController#set_current_user. This makes user data accessible
in models and other non-controller contexts.

4.4 SESSION USER MODEL
-----------------------

File: app/models/session_user.rb
Type: Plain Ruby class (not ActiveRecord)

Purpose: Wraps Employee for authentication and group-checking operations.

Class Methods:
  authenticate(emp_id_or_email) -> SessionUser instance or nil

Instance Methods:
  employee          -> Employee ActiveRecord object
  in_group?(name)   -> Boolean (checks group membership)
  in_any_group?(*names) -> Boolean (checks any group membership)

4.5 ADMIN IMPERSONATION
------------------------

File: app/controllers/admin/impersonations_controller.rb
Routes: /admin/impersonations (new, create, destroy)

Flow:
  1. Admin navigates to impersonation page
  2. Selects employee from list
  3. System stores session[:real_admin_id] = current employee ID
  4. Session replaced with impersonated employee's data
  5. Yellow "Emulating" banner shows in header
  6. Admin can destroy impersonation to restore their own session

Guard: require_super_admin (checks session for admin privileges)

5. AUTHORIZATION & ACCESS CONTROL (ACL)
================================================================================

5.1 GROUP-BASED PERMISSIONS
----------------------------

Groups are stored in the Groups table (PascalCase: GroupID, Group_Name).
Employees are assigned to groups via Employee_Groups join table.
Group permissions are stored in Group_Permissions table.

KEY SYSTEM GROUPS (checked by name, stored downcased):
  - "system_admins"          -> Full admin access (system_admin? helper)
  - "auth_console_admin"     -> Authorization console admin access
  - "auth_console_approvers" -> Authorization console user access
  - "ticket_admin"           -> Help ticket administration
  - "status managers"        -> Submission status management

5.2 PERMISSION TYPES
---------------------

Two permission types exist in Group_Permissions:

DROPDOWN PERMISSIONS (permission_type: 'dropdown'):
  Control visibility of navigation/sidebar items:
  - inbox             -> Inbox queue access
  - submissions       -> Submissions tracker access
  - reports           -> Report generation access
  - dashboards        -> Power BI dashboard access
  - manage_forms      -> Form template builder access
  - emulate           -> Admin impersonation access
  - acl               -> ACL management access
  - auth_console      -> Authorization console access
  - lookup_tables     -> Lookup table management access

FORM PERMISSIONS (permission_type: 'form'):
  Control access to specific restricted form templates.
  Permission_Key = FormTemplate.id

5.3 ORGANIZATION-LEVEL PERMISSIONS
-----------------------------------

Table: org_permissions
Model: OrgPermission

Permissions can also be scoped to organizational hierarchy levels:
  - Agency level    (agency_id only)
  - Division level  (agency_id + division_id)
  - Department level (agency_id + division_id + department_id)
  - Unit level      (agency_id + division_id + department_id + unit_id)

Permission types: 'dropdown' or 'form'
These cascade: a permission at agency level applies to all divisions,
departments, and units within that agency.

5.4 PERMISSION RESOLUTION (ApplicationController)
--------------------------------------------------

current_user_group_names:
  - Memoized Set of downcased group names
  - Built via EmployeeGroup.joins(:group).where(EmployeeID: user_id)
  - Used for system_admin?, auth_console_admin?, etc.

current_user_group_ids:
  - Memoized Array of GroupIDs
  - Used for form permission lookups

current_user_org_chain:
  - Returns hash: { agency_id, division_id, department_id, unit_id }
  - Built from session[:user] data

load_user_permissions(permission_type):
  - Cascades org-level permissions (agency -> division -> department -> unit)
  - Merges with group-level permissions
  - Returns Set of permission keys

current_user_dropdown_permissions:
  - Calls load_user_permissions('dropdown')
  - Controls sidebar navigation visibility

current_user_form_permission_ids:
  - Calls load_user_permissions('form')
  - Controls which restricted forms the user can access

5.5 FORM ACCESS CONTROL
-------------------------

FormTemplate has two access levels:
  - "public"     -> All authenticated employees can access
  - "restricted" -> Access controlled by org_scope OR acl_group_id

Restricted forms check access via FormTemplate#accessible_by?(user_org_chain, user_group_ids):
  - If org_scoped?: checks user's org chain matches org_scope_type/org_scope_id
  - If acl_group_id set: checks user's group IDs include the ACL group

5.6 CONTROLLER-LEVEL GUARDS
-----------------------------

before_action :require_system_admin
  -> Redirects to root_path unless system_admin?
  -> Used by: FormTemplatesController, AclController, ReportsController,
     BillingToolsController, ScheduledReportsController

before_action :require_auth_console
  -> Redirects unless auth_console_admin? or auth_console_user?
  -> Used by: AuthorizationConsoleController

Custom per-controller checks:
  -> ticket_admin? in HelpTicketsController/HelpController
  -> require_authentication in DashboardsController
  -> require_login in TaskReassignmentsController

8. MODELS - FORM TEMPLATE SYSTEM (DYNAMIC FORM BUILDER)
================================================================================

The Form Template system allows administrators to create forms without writing
code. It generates database tables, models, controllers, and views dynamically.

8.1 FORM TEMPLATE
-------------------

File: app/models/form_template.rb
Table: form_templates
Primary Key: id (bigint, auto-increment)

KEY COLUMNS:
  name                    (string, NOT NULL, unique)
  class_name              (string, NOT NULL, unique) - Generated from name
  access_level            (string, default: "public") - "public" or "restricted"
  acl_group_id            (integer, nullable) - FK to Groups for ACL
  page_count              (integer, default: 2, range: 2-30)
  page_headers            (JSON) - Header text per page
  created_by              (integer) - Employee ID of creator
  submission_type         (string, default: "database") - "database" or "approval"
  approval_routing_to     (string, nullable) - Routing strategy
  approval_employee_id    (integer, nullable) - Specific employee for routing
  powerbi_workspace_id    (string, nullable) - Power BI workspace
  powerbi_report_id       (string, nullable) - Power BI report
  has_dashboard           (boolean, default: false)
  inbox_buttons           (JSON array) - Enabled inbox action buttons
  status_transition_mode  (string, default: "automatic") - "automatic" or "manual"
  tags                    (string) - Comma-separated searchable tags
  org_scope_type          (string, nullable) - "agency"/"division"/"department"/"unit"
  org_scope_id            (string, nullable) - ID of scoped org unit

ASSOCIATIONS:
  has_many :form_fields (dependent: :destroy)
  has_many :routing_steps -> FormTemplateRoutingStep (ordered by step_number,
           dependent: :destroy)
  has_many :statuses -> FormTemplateStatus (ordered by position,
           dependent: :destroy)
  accepts_nested_attributes_for :routing_steps, :statuses

VALIDATIONS:
  - name: presence, uniqueness
  - class_name: presence, uniqueness
  - access_level: inclusion in ['public', 'restricted']
  - page_count: 2 to 30
  - org_scope_type: inclusion (allow nil)
  - restricted forms must have scope_or_group
  - submission_type: inclusion in ['database', 'approval']
  - approval_routing_to: required if requires_legacy_routing?
  - approval_employee_id: required if routes_to_specific_employee?
  - powerbi_workspace_id + powerbi_report_id: required if has_dashboard?
  - status_transition_mode: inclusion in ['automatic', 'manual']

SCOPES:
  with_dashboards - where(has_dashboard: true)

KEY METHODS:
  restricted?                -> access_level == 'restricted'
  requires_approval?         -> submission_type == 'approval'
  routes_to_specific_employee? -> requires_approval && routing_to == 'employee'
  has_routing_steps?         -> routing_steps.any? || pending steps present
  has_multiple_routing_steps? -> routing_steps.count > 1
  has_dashboard?             -> has_dashboard == true
  automatic_status_transitions? -> status_transition_mode == 'automatic'
  manual_status_transitions? -> status_transition_mode == 'manual'
  has_inbox_button?(type)    -> Check if button_type in inbox_buttons
  enabled_inbox_buttons      -> Return inbox_buttons array
  table_name                 -> class_name.underscore.pluralize
  file_name                  -> class_name.underscore
  accessible_by?(org_chain, group_ids) -> Check org scope + ACL group
  acl_group_name             -> Get Group_Name of acl_group_id
  page_header(page_num)      -> Get header text for specific page
  tags_array / tags_array=   -> Serialize tags as array
  all_tags (class method)    -> Get all unique tags across templates
  org_scoped?                -> org_scope_type.present? && org_scope_id.present?
  org_scope_label            -> Get long_name of scoped org unit

CONSTANTS:
  INBOX_BUTTON_TYPES - Hash of 7 button types:
    view_pdf, edit, approve, deny, reassign, take_back, status_dropdown
  TRANSITION_MODES - %w[automatic manual]

CALLBACKS:
  before_validation :generate_class_name (on: :create)
  before_validation :clear_restrictions_if_public
  validate :page_count_cannot_orphan_fields (on: :update)

33. KEY ARCHITECTURAL PATTERNS
================================================================================

1. SESSION-BASED AUTH: No Devise/Pundit. All auth via session[:user] hash
   with manual group membership checks. OmniAuth handles Entra ID OAuth.

2. ORG HIERARCHY CASCADE: Agency -> Division -> Department -> Unit.
   Used for: form prefill, permission scoping, approval routing,
   dropdown filtering.

3. POLYMORPHIC TRACKING: StatusChange and TaskReassignment use polymorphic
   belongs_to for form-agnostic status history and reassignment chains.

4. DUAL FORM SYSTEM: Legacy hardcoded forms (3 main workflow forms) coexist
   with dynamically-generated forms via FormTemplate builder. Both appear
   in the same inbox and submissions views.

5. IN-MEMORY FILTERING: Filterable concern operates on Ruby arrays, not
   database queries. This enables cross-model filtering in inbox/submissions
   where data comes from multiple unrelated tables.

6. GENERATED SCAFFOLDING: FormTemplatesController generates complete
   model/controller/view/migration files. Uses MSSQL-specific DDL for
   table creation.

7. CONCERN-BASED COMPOSITION: TrackableStatus, Reassignable, PhoneNumberable
   provide consistent behavior across form models. Models declare which
   concerns they include and implement required abstract methods.

8. PERMISSION CASCADING: Org-level permissions cascade from agency down to
   unit. Group permissions are merged at runtime. Both types are resolved
   in ApplicationController and cached per-request.

9. STATUS CATEGORY NORMALIZATION: Different forms use different status values
   but all map to normalized categories (pending, in_review, approved,
   denied, cancelled, scheduled) for consistent badge display.

10. MULTI-STEP APPROVAL: Some forms have single-step (submit -> approve),
    some have multi-step (submit -> step_1_approved -> step_2_approved),
    and some have delegated approval chains.

  Clicks Approve        -->   approve()               Update status
                              Record status change    (TrackableStatus)
